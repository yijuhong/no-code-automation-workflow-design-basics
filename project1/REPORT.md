# 프로젝트 1 보고서: CS 고객 문의 우선순위 분류 및 알림 자동화

## 1. 프로젝트 개요

이 보고서는 동일한 자동화 워크플로우를 Make와 Zapier로 각각 구현한 뒤, 두 도구의 설정 방식과 사용 경험을 비교한 결과를 정리한다.

- 자동화 주제: CS 고객 문의 우선순위 분류 및 알림 자동화
- 사용한 도구: Make, Zapier
- 공통 구성 요소: Google Form, Google Sheets, Slack
- 알림 채널: Slack `#automation-test`
- 분기 기준: 우선순위 점수 80점 이상은 긴급 문의, 80점 미만은 일반 문의

## 2. 공통 워크플로우 구조

두 도구 모두 같은 흐름으로 구현했다.

1. CS 담당자가 Google Form에 고객 문의를 입력한다.
2. Google Form 응답이 Google Sheets에 새 행으로 저장된다.
3. 자동화 도구가 Google Sheets의 새 행을 감지한다.
4. 우선순위 점수를 기준으로 조건 분기한다.
5. 긴급 문의와 일반 문의에 따라 Google Sheets의 다른 탭에 기록한다.
6. Slack `#automation-test` 채널에 분기별 알림을 전송한다.

분기별 처리 흐름은 다음과 같다.

| 분기 | 조건 | Google Sheets 기록 | Slack 알림 |
| --- | --- | --- | --- |
| 긴급 문의 | 우선순위 점수 80점 이상 | `긴급 문의` 탭에 행 추가 | 긴급 문의 알림 전송 |
| 일반 문의 | 우선순위 점수 80점 미만 | `일반 문의` 탭에 행 추가 | 일반 문의 접수 알림 전송 |

## 3. 도구별 구현 과정 요약

### Make 구현

Make에서는 Google Sheets의 새 행을 감지하는 모듈을 Trigger로 사용했다. 이후 Router와 Filter를 이용해 우선순위 점수 기준으로 긴급 문의와 일반 문의 경로를 나누었다.

각 경로에는 Google Sheets 행 추가 Action과 Slack 메시지 전송 Action을 배치했다. 전체 워크플로우를 즉시 실행할 수 있어 실제 작동 여부를 확인하기 쉬웠다.

- 워크플로우 구성 화면: `project1/make/workflow.png`
- 실행 결과 화면: `project1/make/execution-00.png`, `project1/make/execution-01.png`, `project1/make/execution-02.png`, `project1/make/execution-03.png`, `project1/make/execution-04.png`

### Zapier 구현

Zapier에서는 Google Sheets의 새 행을 감지하는 이벤트를 Trigger로 사용했다. 이후 Paths를 이용해 우선순위 점수 기준으로 긴급 문의와 일반 문의 경로를 나누었다.

각 Path에는 Google Sheets 행 추가 Action과 Slack 메시지 전송 Action을 배치했다. 워크플로우를 설계하는 단계에서 테스트를 즉시 실행할 수 있는 UI가 있어 설정 작업이 직관적이었다.

- 워크플로우 구성 화면: `project1/zapier/workflow.png`
- 실행 결과 화면: `project1/zapier/execution-00.png`, `project1/zapier/execution-01.png`, `project1/zapier/execution-02.png`, `project1/zapier/execution-03.png`, `project1/zapier/execution-04.png`

## 5. 비교 분석

| 비교 항목 | Make | Zapier |
| --- | --- | --- |
| UI/UX | 워크플로우 전체를 실행해 실제 흐름을 확인하기 쉬웠다. | 설계 단계에서 단계별 테스트를 바로 실행할 수 있어 설정 과정이 직관적이었다. |
| 설정 난이도 | 개별 테스트 실행이 부족해 확인할 때마다 전체 워크플로우를 실행해야 했다. | 단계별 테스트 UI가 있어 설정 중 확인 작업이 수월했다. |
| 조건 분기 방식 | Router와 Filter를 이용해 긴급 문의와 일반 문의 경로를 나누었다. | Paths를 이용해 긴급 문의와 일반 문의 경로를 나누었다. 다만 Path가 한쪽 분기 위주로 진행되는 UI라 다른 분기를 구성할 때 어려움이 있었다. |
| 연동 서비스 | 이번 실습 범위에서는 Google Sheets와 Slack 연동 구성에서 Zapier와 큰 차이를 느끼지 않았다. | 이번 실습 범위에서는 Google Sheets와 Slack 연동 구성에서 Make와 큰 차이를 느끼지 않았다. |
| 무료 또는 trial 범위 | 유료 결제 없이 무료 또는 trial 범위에서 실습을 완료했다. | 유료 결제 없이 무료 또는 trial 범위에서 실습을 완료했다. |
| 실행 로그 | 전체 워크플로우 실행 결과를 통해 실제 작동 여부를 확인했다. | 각 단계별 실제 API 호출과 응답 내용을 확인할 수 있어 서비스 개발 관점의 디버깅에 더 용이해 보였다. |
| 유지보수성 | 이번 실습 범위에서는 Zapier와 큰 차이를 느끼지 않았다. | 이번 실습 범위에서는 Make와 큰 차이를 느끼지 않았지만, 단계별 테스트와 API 호출/응답 확인은 수정과 디버깅에 도움이 될 것으로 판단했다. |

## 6. 장단점 정리

| 도구 | 장점 | 단점 |
| --- | --- | --- |
| Make | 전체 워크플로우를 즉시 실행할 수 있어 실제 작동 확인이 쉬웠다. | 개별 테스트 실행이 없어 매번 전체 워크플로우를 실행해야 했다. |
| Zapier | 설계 단계에서 테스트를 바로 실행할 수 있는 직관적인 UI가 있어 작업이 용이했다. 단계별 실제 API 호출과 응답 내용을 확인할 수 있어 디버깅에도 유리해 보였다. | Paths에서 한쪽 분기만 타도록 진행되는 UI 때문에 다른 분기를 구성할 때 어려움이 있었고, 이 과정에서 시간이 많이 소요되었다. |

## 7. 적합한 상황

Make는 전체 워크플로우를 빠르게 실행하면서 실제 자동화가 동작하는지 확인해야 하는 상황에 적합하다.

Zapier는 자동화 도구를 처음 사용하는 상황이나, 설계 단계에서 단계별 테스트와 API 호출/응답 확인을 통해 디버깅하면서 구성해야 하는 상황에 더 적합하다.

## 8. 최종 의견

이번 프로젝트에서는 Zapier가 더 적합하다고 판단했다. Make는 전체 워크플로우를 즉시 실행해 실제 동작을 확인하기 쉬웠지만, 개별 단계 테스트가 부족해 반복 확인 과정에서 불편함이 있었다.

반면 Zapier는 설계 단계에서 테스트를 바로 실행할 수 있는 직관적인 UI를 제공했고, 각 단계의 실제 API 호출과 응답 내용을 확인할 수 있어 문제를 파악하기 쉬웠다. Paths 분기 구성에서 어려움은 있었지만, 전체적인 설정과 디버깅 경험을 기준으로는 Zapier가 더 우세했다.
